# Лабораторная работа №2. Проектирование структуры клиентской части Web-приложения (HTML + архитектурное мышление)

---

## 1. Теория

### 1.0. Цель работы

Сформировать у обучающегося системное понимание HTML как языка разметки клиентской части Web-приложения:

* что такое HTML и зачем он нужен в Web-архитектуре;
* как устроен HTML-документ и почему структура важнее “красоты”;
* как элементы HTML образуют DOM-дерево;
* как проектировать структуру интерфейса под сценарии и будущую интеграцию с API;
* как писать валидную, семантическую, доступную разметку.

Закрепить навык архитектурного описания UI через схему:

```
UI (HTML) → DOM → JS (events) → API (HTTP) → DB
```

и понимание того, что клиентская разметка не существует отдельно, а проектируется под данные и сценарии взаимодействия.

---

### 1.1. Ключевые понятия: HTML, DOM, браузер, стандарты

#### HTML (HyperText Markup Language)

HTML — язык разметки гипертекста, используемый для описания структуры документа и интерфейса в браузере.
В архитектуре Web-приложения HTML выполняет роль структурного слоя:

* задаёт “скелет” интерфейса (разделы, блоки, формы, таблицы, навигацию);
* формирует DOM-дерево, с которым далее работает JavaScript;
* создаёт смысловую структуру (семантику), важную для доступности и SEO.

Важно: HTML не отвечает за логику (это JS) и не отвечает за внешний вид (это CSS), но определяет основу, на которой строится всё остальное.

---

#### DOM (Document Object Model)

DOM — объектная модель документа: браузер превращает HTML-код в дерево узлов, где каждый элемент становится объектом.

Упрощённо:

```
HTML-текст → парсер → DOM-дерево → рендер (layout/paint)
```

DOM нужен для того, чтобы:

* находить элементы (`querySelector`);
* менять содержимое (`textContent`, `innerHTML`);
* добавлять/удалять узлы (`createElement`, `append`);
* реагировать на события пользователя (`click`, `submit`, `input`).

---

#### Браузер как интерпретатор HTML

Браузер выполняет технологическую цепочку:

1. парсит HTML → строит DOM
2. загружает CSS → строит CSSOM
3. объединяет DOM + CSSOM → render tree
4. layout → paint → compositing

Вывод для практики: структура HTML напрямую влияет на:

* поддерживаемость (легко ли читать и развивать проект);
* доступность (может ли интерфейс читать скрин-ридер);
* SEO (понимают ли роботы структуру страницы);
* производительность (например, избыточные вложенности усложняют рендер).

---

### 1.2. История и стандарты HTML

#### Эволюция HTML (HTML 2.0 → HTML5)

HTML развивался от простого языка разметки документов до стандарта, поддерживающего полноценные web-приложения.

* HTML 2.0 (1995) — базовые теги: текст, ссылки, формы (примитивно).
* HTML 3.2 / 4.01 — таблицы, скрипты, стили, но часто без семантики (много “разметки ради внешнего вида”).
* XHTML (2000-е) — попытка сделать HTML “строгим как XML”.
* HTML5 (с 2014+) — семантические теги, медиа, новые формы, стандартизация API браузера.

---

#### Роль W3C и WHATWG

* W3C — консорциум, который исторически утверждал версии спецификаций.
* WHATWG — сообщество, которое ведёт HTML как “Living Standard” (живой стандарт, обновляется постоянно).

Практический вывод: современные браузеры ориентируются на актуальную спецификацию HTML Living Standard.

---

#### Отличие HTML от XML, XHTML

| Критерий | HTML | XML | XHTML |
|---|---|---|
| Назначение | разметка интерфейса/страниц | универсальные данные | HTML в правилах XML |
| Ошибки | браузер “прощает” многое | строгий синтаксис | строгий синтаксис |
| Закрывающие теги | иногда опциональны | обязательны | обязательны |
| Где используется | UI в браузере | интеграции, конфиги, данные | почти не используется в новом web |

---

### 1.3. Структура HTML-документа

#### Базовые элементы документа

* `<!DOCTYPE html>` — сообщает браузеру режим стандартов.
* `<html>` — корневой элемент.
* `<head>` — метаданные, подключение ресурсов.
* `<body>` — видимое содержимое.

---

#### Кодировка и метатеги

* `charset="UTF-8"` — стандартная кодировка.
* `viewport` — корректная адаптивность на мобильных.

---

#### Подключение ресурсов

* CSS: `<link rel="stylesheet" href="...">`
* JS: `<script src="..." defer></script>`

---

#### Пример базового документа с комментариями

```html
<!doctype html>
<html lang="ru">
  <head>
    <!-- Кодировка -->
    <meta charset="UTF-8" />

    <!-- Адаптивность -->
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />

    <!-- Заголовок вкладки -->
    <title>Каталог товаров</title>

    <!-- Описание для SEO -->
    <meta name="description" content="Каталог товаров: поиск, фильтры, карточки." />

    <!-- Подключение CSS -->
    <link rel="stylesheet" href="./styles.css" />

    <!-- Подключение JS (defer — после построения DOM) -->
    <script src="./app.js" defer></script>
  </head>

  <body>
    <header>
      <h1>Каталог</h1>
    </header>

    <main>
      <p>Контент страницы…</p>
    </main>

    <footer>
      <small>© 2026</small>
    </footer>
  </body>
</html>
```

## 2. Задание

### Цель

Спроектировать структуру клиентской части Web-приложения на уровне HTML, описав её как систему:

```
UI (HTML) → DOM → JS (events) → API (HTTP) → DB
```

Показать, что HTML в проекте:

* описывает интерфейсные блоки (layout);
* задаёт “контракт” для JS (id/class/data-атрибуты);
* формирует основу для отправки данных на сервер (формы);
* обеспечивает семантику, доступность и SEO.

---

### 2.1. Выбор объекта проектирования

Выбрать предметную область для интерфейса Web-приложения:

* интернет-магазин;
* LMS (платформа курсов);
* сервис бронирования;
* ServiceDesk / заявки;
* доставка / логистика;
* расписание / запись к специалисту.

Требование: интерфейс должен предполагать взаимодействие с данными (каталог, формы, карточки, статусы), а не быть статической страницей “о компании”.

---

### 2.2. Аналитический разбор интерфейса (UI-структура)

В отчёте указать:

1. Основной сценарий пользователя (например: “выбрать товар → заполнить форму → отправить”).
2. Какие экранные области необходимы:

   * шапка/навигация;
   * контентная область;
   * фильтры/панели;
   * карточки/списки;
   * формы.

3. Какие типы данных будут отображаться (примерно):

   * список сущностей (товары/курсы/заявки);
   * детальная карточка;
   * параметры фильтрации;
   * данные формы.

4. Какие элементы HTML подходят под каждую область (семантика).

---

### 2.3. Архитектурная схема “UI → API → DB”

Построить блок-схему (упрощённо):

```
User → UI (HTML) → DOM → JS → API → DB → API → JS → UI update
```

Обязательно показать:

* пользователь;
* интерфейс (HTML/DOM);
* клиентскую логику (JS);
* серверный API;
* базу данных.

---

### 2.4. Структура HTML (каркас будущего интерфейса)

Создать HTML-страницу с обязательной структурой:

* `<header>` + `<nav>`
* `<main>` (основное содержимое)
* `<aside>` (дополнительная панель: фильтры/корзина/статусы — по теме)
* `<footer>`

Обязательно использовать:

* минимум 1 `section`;
* минимум 2 `article` (карточки сущностей);
* заголовочную иерархию `h1 → h2 → h3`.

---

### 2.5. Таблица сравнения: блочные / строчные / void / семантические

Составить таблицу (минимум 12 элементов), где для каждого элемента указать:

* тип (block/inline/void/semantic);
* назначение;
* пример использования (1 строка кода).

Пример формы таблицы (заполнить своими примерами):

| Элемент   | Тип            | Назначение          | Пример                       |
| --------- | -------------- | ------------------- | ---------------------------- |
| `section` | semantic block | тематический раздел | `<section>...</section>`     |
| `a`       | inline         | ссылка/навигация    | `<a href="/catalog">...</a>` |
| `img`     | void           | изображение         | `<img src="..." alt="...">`  |

---

### 2.6. Проектирование формы (минимум 1 форма)

В интерфейсе должна быть форма, ориентированная на работу с Web-сервисом:

* обязательные поля (`required`);
* корректные типы (`email`, `number`, `date`, `url` — по необходимости);
* минимум 1 `pattern`;
* минимум 1 `select` или группа `radio`.

Форма должна быть “API-ready”, то есть:

* поля имеют `name`;
* есть `id` + `label for`;
* структура понятна для отправки на сервер.

---

### 2.7. Передача данных на API (демонстрационный сценарий)

Добавить пример отправки формы через `fetch()` (как демонстрация связи UI → API).
В отчёте указать:

* URL;
* HTTP-метод;
* заголовки;
* пример JSON-тела;
* ожидаемый статус-код;
* пример JSON-ответа.

---

### 2.8. Краткий аналитический вывод

1–2 страницы:

* почему выбранная структура HTML корректна с точки зрения семантики;
* как DOM будет использоваться JS (какие узлы будут точками привязки);
* как форма связана с REST-сценарием;
* какие потенциальные ошибки предотвращены (доступность, валидность, id/name).
